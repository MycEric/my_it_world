# My IT World — 个人展示网站系统设计企划书

> 版本：v1.2  
> 技术栈：React（前端）+ Spring Cloud Alibaba（Java 微服务）+ Python（AI 服务）  
> 目标：构建一个可展示个人能力、博客文章，支持用户登录，并可通过 AI Agent 基于博客知识库进行智能问答的个人品牌网站  
> 变更说明（v1.2）：补充产品功能全景、实现状态矩阵、非 AI 待设计模块（搜索/互动/运营/站点配置）及分阶段优先级

---

## 1. 项目概述

### 1.1 项目背景

本项目旨在搭建一个**个人 IT 学习成果展示平台**，用于集中呈现：

- 个人简介与职业形象
- 技术能力与项目经历
- 在 CSDN 等平台发布的博客文章
- 基础的用户体系（访客浏览 + 注册用户/管理员）
- **（后期）AI 知识助手**：将博客等内容向量化，用户提问时通过 RAG + Agent 检索并流式回答

### 1.2 项目目标

| 维度 | 目标 |
|------|------|
| 学习价值 | 完整实践 Java 微服务架构（注册中心、网关、鉴权、服务拆分、分布式缓存等） |
| 展示价值 | 对外呈现个人技术栈与产出，可作为简历/作品集入口 |
| 可扩展性 | 模块解耦，后续可接入 GitHub、LeetCode、RSS、**Python AI 服务（RAG/Agent）** 等 |
| AI 能力 | 博客内容 Embedding 入库，访客/用户可针对站内知识库进行自然语言问答（流式输出） |
| 可运维性 | 容器化部署，支持本地开发与生产环境切换 |

### 1.3 用户角色

| 角色 | 权限说明 |
|------|----------|
| 访客（Guest） | 浏览首页、关于我、技能、项目、博客列表与详情；**使用 AI 助手提问（可限流）** |
| 注册用户（User） | 访客权限 + 收藏文章、评论（可选）；**AI 对话历史保存（可选）** |
| 管理员（Admin） | 全部权限 + 后台管理个人信息、技能、项目、触发 CSDN 同步；**上传/触发博客向量化、管理知识库** |

### 1.4 产品功能地图（PM 视角）

除 **AI Agent + RAG** 外，完整产品应覆盖「展示 → 内容 → 发现 → 互动 → 运营 → 基础设施」六层能力。当前实现与待设计模块如下：

```
┌─────────────────────────────────────────────────────────────────────────┐
│  展示层    首页聚合 │ 关于我 │ 技能 │ 项目 │ 时间线          【待建设】 │
├─────────────────────────────────────────────────────────────────────────┤
│  内容层    博客 CRUD │ CSDN 同步 │ Markdown │ 图片资产        【部分完成】│
├─────────────────────────────────────────────────────────────────────────┤
│  发现层    列表分页 │ 分类 │ 全文搜索 │ 相关推荐 │ 标签       【待建设】 │
├─────────────────────────────────────────────────────────────────────────┤
│  互动层    评论 │ 收藏 │ 点赞 │ 联系表单 │ 邮件订阅          【待设计】 │
├─────────────────────────────────────────────────────────────────────────┤
│  AI 层     简单对话 │ 会话历史 │ RAG │ Agent │ 知识库管理   【一期完成】│
├─────────────────────────────────────────────────────────────────────────┤
│  运营层    数据看板 │ 访问统计 │ AI 用量 │ 同步日志 │ 审计   【待设计】 │
├─────────────────────────────────────────────────────────────────────────┤
│  基础层    鉴权 │ 网关 │ 文件 │ 用户资料 │ 站点配置 │ 运维    【部分完成】│
└─────────────────────────────────────────────────────────────────────────┘
```

**结论（产品优先级建议）**：

1. **P0 — 个人品牌闭环**：content-service（关于/技能/项目）+ 首页聚合 + file-service（头像/封面）— 访客打开站点能完整了解「你是谁、会什么、做过什么」。  
2. **P1 — 内容增长**：博客 RSS/定时同步 + 全文搜索 — 博客可持续更新且可被找到。  
3. **P2 — AI 差异化**：RAG + SSE + 知识库 Admin — 与博客内容联动的智能问答。  
4. **P3 — 社区与运营**：评论/收藏、访问统计、管理看板 — 提升粘性与可观测性。

### 3.2 模块一：API 网关（gateway-service）

**职责**

- 所有外部请求的唯一入口
- 路由到各微服务
- JWT Token 校验（白名单放行登录/注册/公开内容）
- 跨域 CORS 处理
- 统一响应格式与异常转换
- Sentinel 限流（如登录接口防刷）

**路由规划示例**

| 路径前缀 | 目标服务 | 是否鉴权 |
|----------|----------|----------|
| `/api/auth/**` | auth-service | 部分公开 |
| `/api/users/**` | user-service | 是 |
| `/api/content/**` | content-service | 读公开，写需 Admin |
| `/api/blogs/**` | blog-service | 读公开，同步需 Admin |
| `/api/files/**` | file-service | 上传需登录 |
| `/api/ai/**` | ai-service (Python) | 问答只读可公开（限流）；入库/管理需 Admin |

> **混合架构说明**：Java 微服务负责业务 CRUD 与鉴权；Python 服务专注 AI 管线（Embedding、检索、Agent、流式 LLM）。两者通过 **Gateway 统一入口** 对外，内部通过 HTTP / 消息队列通信。

### 3.3 模块二：认证服务（auth-service）

**职责**

- 用户注册、登录、登出
- JWT Access Token + Refresh Token 签发与刷新
- 密码加密（BCrypt）
- 角色权限（RBAC）基础能力
- 登录日志记录

**核心功能**

| 功能 | 接口 | 说明 |
|------|------|------|
| 注册 | `POST /api/auth/register` | 邮箱/用户名注册 |
| 登录 | `POST /api/auth/login` | 返回 Token |
| 刷新 Token | `POST /api/auth/refresh` | Refresh Token 换新 Access Token |
| 登出 | `POST /api/auth/logout` | Token 加入 Redis 黑名单 |
| 获取当前用户 | `GET /api/auth/me` | 解析 Token 返回用户信息 |

**JWT 设计**

```
Access Token  : 有效期 2h，携带 userId、roles
Refresh Token : 有效期 7d，仅存 Redis，支持吊销
```

**数据库表（auth 库）**

- `sys_user` — 用户账号
- `sys_role` — 角色（ADMIN、USER）
- `sys_user_role` — 用户角色关联
- `sys_login_log` — 登录日志

### 3.6 模块五：博客服务（blog-service）— CSDN 文章同步

**职责**

- 从 CSDN 拉取/同步个人博客文章
- 本地存储文章元数据与正文（缓存）
- 提供搜索、分类、分页列表与详情

**CSDN 同步方案（推荐两种，可组合）**

#### 方案 A：RSS 订阅（推荐优先实现）

- CSDN 个人主页通常提供 RSS：`https://blog.csdn.net/{username}/rss/list`
- 定时任务（Spring `@Scheduled` 或 XXL-JOB）拉取 RSS
- 解析 XML 得到标题、链接、摘要、发布时间
- **优点**：稳定、合规、实现简单  
- **缺点**：通常只有摘要，无完整正文

#### 方案 B：CSDN 开放 API / 页面解析（进阶）

- 调研 CSDN 是否提供官方 API（需申请）
- 若无官方 API，可抓取文章详情页 HTML 解析正文（注意合规与反爬）
- 建议：**仅同步自己的文章**，控制频率，遵守 robots.txt

#### 方案 C：手动导入 + 半自动同步（稳妥）

- 管理后台支持 Markdown 手动录入
- RSS 自动同步元数据，正文首次访问时懒加载并持久化

**同步流程**

```
定时任务触发
    → 拉取 RSS/API
    → 对比本地 article.source_id
    → 新增/更新文章
    → （可选）发送 RocketMQ 消息异步拉取正文
    → 写入 MySQL + 缓存 Redis
```

**核心接口**

| 功能 | 接口 |
|------|------|
| 文章分页列表 | `GET /api/blogs?page&size&keyword&category` |
| 文章详情 | `GET /api/blogs/{id}` |
| 热门/最新 | `GET /api/blogs/latest?limit=5` |
| 手动触发同步 | `POST /api/blogs/sync`（Admin） |
| 管理端编辑 | `PUT /api/blogs/admin/{id}` |

**数据库表（blog 库）**

- `blog_article` — id, title, summary, content, cover, source(CSDN), source_id, source_url, publish_time, view_count, status
- `blog_category` — 分类
- `blog_article_category` — 多对多
- `blog_sync_log` — 同步记录（时间、成功数、失败原因）

**缓存策略**

- 列表：Redis，TTL 10min，同步后主动失效
- 详情：Redis，TTL 30min，按浏览量可延长

### 3.8 模块七：AI 知识助手服务（ai-service — Python）

**定位**

独立的 **Python 微服务**，与 Java 微服务并列，专门负责：

1. **知识入库（Ingest）**：接收博客/文档内容 → 切分 Chunk → Embedding → 写入向量数据库  
2. **智能问答（Chat）**：用户提问 → 向量检索相关博客片段 → Agent 组装上下文 → 调用大模型 **流式** 回答  
3. **知识库管理（Admin）**：触发重建索引、查看入库状态、删除/更新某篇文章的向量

**为什么用 Python 而不是 Java？**

| 考量 | 说明 |
|------|------|
| AI 生态 | LangChain、LlamaIndex、各类 Embedding/LLM SDK 以 Python 为主 |
| 流式输出 | FastAPI + SSE 实现简单，社区范例丰富 |
| 职责分离 | Java 管业务，Python 管 AI，互不阻塞，可独立扩缩容 |
| 学习价值 | 实践「多语言微服务 + RAG + Agent」完整链路 |

**核心流程一：博客向量化入库（Ingest Pipeline）**

```
触发源（三选一或组合）
  ├─ blog-service 文章新增/更新 → RocketMQ / HTTP 回调 ai-service
  ├─ Admin 后台手动「同步到知识库」
  └─ Admin 上传 Markdown/文本文件

        ↓
ai-service 接收文档
        ↓
文档预处理（去 HTML 标签、保留代码块、提取标题层级）
        ↓
文本切分 Chunk（RecursiveCharacterTextSplitter，512~1024 token，overlap 128）
        ↓
调用 Embedding 模型 → 得到 float[] 向量
        ↓
写入向量库（含 metadata：blog_id、title、chunk_index、source_url、publish_time）
        ↓
MySQL 记录索引状态 ai_document_index（status=done，chunk_count=N）
```

**核心流程二：RAG + Agent 问答（Chat Pipeline）**

```
用户在前端 AI 助手页输入问题
        ↓
POST /api/ai/chat （Gateway 鉴权 + 限流）
        ↓
ai-service Agent：
  1. 对用户问题做 Embedding
  2. 向量库 Top-K 相似检索（K=5~10，可加重排序 rerank）
  3. 组装 Prompt（系统提示 + 检索到的博客片段 + 用户问题）
  4. 调用 LLM stream=True
        ↓
SSE 流式返回：event: token / event: source / event: done
        ↓
前端逐字渲染 Markdown，并展示「引用来源」（跳转到对应博客）
```

### 3.9 模块八：搜索与内容发现（search — 建议首期并入 blog-service）

**产品价值**：博客数量上百篇后，仅靠列表分页无法满足「找文章」需求；AI RAG 解决「问知识」，搜索解决「精确找标题/关键词」。

**职责**

- 博客全文检索（标题、摘要、正文）
- 分类/标签聚合、热门排行

**技术选型（二选一）**

| 方案 | 适用 | 说明 |
|------|------|------|
| MySQL FULLTEXT | 文章 < 5000，快速落地 | 零额外组件，中文需 ngram 配置 |
| Elasticsearch 8.x | 长期、高并发 | 与阶段五「搜索增强」一致，独立索引 |

**核心接口**

| 功能 | 接口 | 权限 |
|------|------|------|
| 综合搜索 | `GET /api/blogs/search?q=&page&size` | 公开 |
| 热门文章 | `GET /api/blogs/hot?days=7&limit=10` | 公开 |
| 相关文章 | `GET /api/blogs/{id}/related?limit=5` | 公开 |
| 标签云 | `GET /api/blogs/tags` | 公开 |
| 重建搜索索引 | `POST /api/blogs/admin/search/reindex` | Admin |

**数据库扩展**

- `blog_article` 增加 `view_count` 累计（已有字段则复用）、`search_vector`（若用 ES 则不必）
- `blog_tag`、`blog_article_tag`（若引入标签体系）

**与 AI 的关系**：RAG 面向自然语言问答；搜索面向明确关键词。产品应在博客列表页提供搜索框，在 AI 回答中引用来源跳转详情。

## 7. 安全设计

| 兼容性 | Chrome / Edge / Firefox 最新两个大版本 |
| SEO | 博客详情可考虑 SSR/SSG（Next.js 迁移，三期可选） |
| **AI 首 token 延迟** | < 3s（含检索 + LLM 冷启动） |

## 12. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| CSDN RSS 变更或限流 | 同步失败 | 保留手动导入；日志告警；降低抓取频率 |
| 微服务过度拆分 | 开发复杂度高 | 初期可合并 auth+user，稳定后再拆 |
| JWT 无法主动失效 | 安全风险 | Refresh Token 存 Redis；登出加黑名单 |
| 单机资源不足 | 本地跑不动 | Docker 限制内存；非必要服务按需启动 |
| **Embedding/LLM API 费用** | 成本超支 | 本地 BGE 模型 + 问答限流 + max_tokens 限制 |
| **向量库与 MySQL 不一致** | 问答引用过期内容 | 博客更新发 MQ 重建索引；rebuild 兜底任务 |
| **Python 与 Java 异构部署** | 运维复杂度 | Docker 统一编排；Gateway 统一入口；独立扩缩 ai-service |
| **SSE 长连接被 Nginx 缓冲** | 流式失效 | Nginx 配置 `proxy_buffering off`；`X-Accel-Buffering: no` |

已完成：鉴权 + 博客 + AI 对话会话
